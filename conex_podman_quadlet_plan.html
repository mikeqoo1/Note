一、目標

建立一套適合企業部署、可重複安裝、可維護、可升級的 Conex Platform 容器化架構。

  • 採用 Podman，而非 Docker。
  • 使用 systemd 管理所有服務生命週期。
  • 核心資料庫採 PostgreSQL 18 + TimescaleDB。
  • 容器盡量採 Rootless 模式，降低權限風險。
  • 支援一鍵部署、備份、升級與還原。
  • 資料與程式完全分離。
  • 主要適用 Rocky Linux 9 / AlmaLinux 9。

二、目前規劃架構

Systemd (PID 1) │ ▼ Quadlet │ ▼ Podman │ ▼ Container

預計所有服務皆由 Quadlet 管理，例如 PostgreSQL、Redis、RabbitMQ、Grafana、Prometheus、SonarQube、API Server、Worker 等。

三、目錄規劃

系統設定

/etc/containers/systemd/

放置所有 system-level Quadlet：

conex-postgres.container
conex-redis.container
conex-grafana.container
...

安裝程式

/opt/conex/
├── install.sh
├── upgrade.sh
├── backup.sh
├── images/
│   ├── postgres/
│   └── redis/
└── scripts/

資料目錄

/data/conex/
├── postgres/
├── redis/
├── grafana/
├── backup/
└── logs/
原則：程式、Image 原始碼、systemd 設定、資料路徑分離，方便升級、備份與搬遷。

四、Image 規劃

不直接使用 PostgreSQL 官方 Image 作為最終正式環境，而是採用自建 Image。

postgres:18-trixie
        +
TimescaleDB
        +
Conex 初始化設定

範例 Dockerfile 概念：

FROM postgres:18-trixie

# 安裝 TimescaleDB
# COPY initdb/ /docker-entrypoint-initdb.d/

Image 命名範例：

localhost/conex-postgres:18-trixie-timescaledb
Quadlet 只負責執行容器，不負責在容器啟動時安裝套件。套件安裝應在 Image build 階段完成。

五、Quadlet 規劃

範例檔案：

/etc/containers/systemd/conex-postgres.container

Container 區段主要負責：

  • Image
  • Port
  • Volume
  • Environment
  • Restart Policy

Service 區段可設定：

[Service]
Restart=always
RestartSec=10
TimeoutStartSec=300

六、Rootless 架構討論

方案 位置 操作方式 特性
User Quadlet ~/.config/containers/systemd/ systemctl --user 官方推薦的 Rootless 模式，但管理較分散，需要 linger。
System Quadlet /etc/containers/systemd/ systemctl 官方成熟模式，集中管理，但預設為 rootful。
System Quadlet + User= /etc/containers/systemd/ systemctl 目標是 systemd 集中管理，但 Podman 以服務帳號 rootless 執行。官方語法可行，但需自行驗證。

方案三流程

root systemd │ ▼ conex-postgres.service │ ▼ User=postgres │ ▼ rootless podman │ ▼ postgres container

Quadlet 的 [Service] 區段可以加入：

[Service]
User=postgres
Group=postgres
目前官方文件允許 [Service] 選項傳給 systemd，因此語法可行；但官方 Rootless 範例主要仍是 user systemd 模式，所以此方案需做 PoC 驗證後再列為公司標準。

七、目前採用方向

目前傾向採用：

System Quadlet
+
Rootless Podman
+
User=Service Account

原因：

  • 服務由 systemd 集中管理。
  • 容器以 rootless 執行，降低權限風險。
  • 較符合企業主機統一維護方式。

八、PostgreSQL 規劃

Image localhost/conex-postgres:18-trixie-timescaledb
資料 /data/conex/postgres/data
備份 /data/conex/postgres/backup
初始化 SQL /docker-entrypoint-initdb.d/

TimescaleDB 初始化：

CREATE EXTENSION IF NOT EXISTS timescaledb;

十、待驗證事項（PoC）

  1. Rootless 驗證：確認 podman info 是否顯示 rootless: true。
  2. 開機自啟：確認 systemctl enable 後，重新開機容器可正常啟動。
  3. Linger：確認 loginctl enable-linger postgres 是否仍為必要條件。
  4. Quadlet + User=：確認 [Service] User= 與 Group= 在 Podman 6.x 行為穩定。
  5. Auto Update：驗證 AutoUpdate=registry 在 rootless 模式是否正常。
  6. SELinux：驗證 Volume 使用 :Z 是否即可正常運作，是否需額外 policy。
  7. PostgreSQL Volume：確認 PG18 掛載 /var/lib/postgresql 是否符合官方建議。
  8. Backup：建立並驗證 backup.sh，包含 pg_dump、pg_basebackup、WAL Archive。
  9. Restore：驗證全新主機 Restore 後可正常啟動。
  10. Upgrade：驗證 PostgreSQL 18.x 與未來 19 的升級流程。
  11. TimescaleDB：驗證 ALTER EXTENSION 升級流程。
  12. Log：確認 journalctl 是否足以取代日常 container log 管理。
  13. Monitoring：驗證 Cockpit 是否可正常管理 Quadlet、systemd 與容器狀態。
